# FitCree デザイン・開発ワークフローガイド (AI-First Approach)

## 1. コンセプト：Figmaを使わないデザイン開発

FitCreeの開発では、Figma等のデザインツールによるプロトタイピングを最小限にし、Cursor（AI）とコード（Tailwind CSS / React）によるダイレクトなデザイン実装を採用します。

### なぜこの手法をとるのか

- **スピード**: デザインツールからコードへの変換プロセスを排除し、即座にVercel等で実機確認を行うため。
- **一貫性**: 共通のUIキットと仕様書（Spec）をソースオブトゥルース（信頼できる唯一の情報源）とし、デザインのズレを防止するため。
- **SDD（仕様駆動開発）**: UIの変更を「コードの修正」ではなく「仕様データの修正」として扱うことで、非エンジニアでもデザイン調整を可能にするため。

## 2. デザインの3要素（ソースファイル）

Cursorはこの3つのファイルを常に参照し、一貫性を保ちます。

### カラー定義 (`Colors.md`)

- `orange-creator`, `blue-client` などのブランドカラー定義。
- 背景色（`slate-100`）やテキスト色の階層定義。

### UIキット (`FormElements.tsx` / `FormLayout.tsx`)

- ボタン、入力フォーム、カード、ステッパー等の「部品」を定義。
- **ルール**: AIは新しい部品をゼロから作らず、可能な限りここにあるコンポーネントを再利用すること。

### 仕様書 (`job-post-spec.ts`)

- 画面に表示する項目、順序、バリデーション等の「設計図」。
- **ルール**: 画面構成の変更は、このファイルを書き換えることで行う。

## 3. ワークフロー：AIへの指示の出し方

非エンジニアメンバーがAIを使ってデザインを調整する際の標準的な手順です。

### 手順 1: 仕様の追加・変更

AIに対し、まず「何を表示したいか」を指示します。

**指示例**: 「案件投稿のStep 3に、ファイルアップロード機能を追加して。ラベルは『参考資料』で、PDFとJPGを許可するように仕様書を更新してください。」

### 手順 2: レイアウト・見た目の調整

UIキットの範囲内で見た目を調整させます。

**指示例**: 「Step 1の業界選択（CheckboxGrid）を、今の3カラムから4カラムに変更して。」

**指示例**: 「全体の背景を `Colors.md` にある `slate-100` に統一し、各フォームセクションに白いカード（`bg-white`）と軽い影（`shadow-sm`）をつけてください。」

### 手順 3: プレビューと微調整

ブラウザで確認し、違和感があればAIに修正を依頼します。

**指示例**: 「ボタンの文字が少し小さいので、`text-sm` から `text-base` に変更して、フォントを太字（`font-bold`）にしてください。」

## 4. Cursor (AI) への重要指示事項

AIがこのファイルを読み込んだ際、守るべき原則です。

- **色の遵守**: `Colors.md` で定義されたクラス名以外の色を勝手に生成しないでください。
- **コンポーネント優先**: UIの変更依頼があった際、まずは `FormElements.tsx` に既存のパーツがないか確認し、あればそれを使用、なければ `FormElements.tsx` に新しい汎用パーツとして追加してから画面に反映してください。
- **仕様駆動**: ページファイルのハードコーディングを避け、`specs/` ディレクトリ内の定義ファイルに基づいた動的なレンダリングを維持してください。
- **アクセシビリティの遵守**: 実装時は必ず `/docs/rules/accessibility.md` を読み込み、定義された基準（WCAG A/AA）をクリアしたコードを生成してください。

## 5. 運用上の注意

- **実機至上主義**: Cursorで修正し、GitHubにプッシュしてVercelで確認したものが「完成形」です。
- **フィードバックループ**: メンバーがAIを使ってデザインを変えた場合、その意図（例：なぜ4カラムにしたのか）をコメントとしてコードやドキュメントに残すようAIに指示してください。
